# bugs
In general all bugs refer to TGTT classes unless otherwise specified.

## Open Bugs
[] when exporting characters as a json, we get a file called "name.json.json" instead of "name.json". This is a minor issue but should be fixed for better user experience.
[] the Dreamer feat from the Traveler's Guide to Thelemar isn't granting the expected benefits, and may not be fully implemented in calculations.
[] Nyuidj from TGTT are supposed to have an advantage in Wis Saving Throws, and they do! But when trying to roll with disadvantage, it just rolls with disadvantage instead of normal. This is a bug in the roll logic that needs to be fixed to properly handle cases where a character has both advantage and disadvantage on the same roll.

I will add the json of the character to the bug report, but basically the cantrips known and cantrips prepared are not being calculated correctly, which leads to some weird issues with spellcasting and spell preparation.
[] Not all warlock invocations are implemented in calculations, and many have prerequisites that are not enforced in the UI (e.g. invocations that require a certain pact boon or warlock level). 

## Closed Bugs
[x] Chronourgy Wizard wasn't getting +INT mod to initiative from Temporal Awareness — calc layer was correct (`getFeatureCalculations()` set `hasTemporalAwareness=true` and `temporalAwarenessBonus=intMod` for Chronurgy L2) but no consumer ever read those flags. Investigation surfaced the same gap for Gloom Stalker's Dread Ambusher (+WIS) and 2024 Champion's Remarkable Athlete (+PB), and a related freshness bug in Swashbuckler's Rakish Audacity (+CHA) which was wired through the auto-effect pipeline → `customModifiers.initiative`, freezing the CHA mod at ingestion time so ASI changes never propagated. Fixed once, generically, by adding a small live aggregator `_getFeatureInitiativeBonuses()` on `CharacterSheetState` that reads the existing calculation flags and returns `[{name, value}]` for Temporal Awareness, Dread Ambusher, Rakish Audacity, and Remarkable Athlete; both `getInitiative()` and `getInitiativeBreakdown()` now consume it (the breakdown pushes one ⚡ component per entry, picked up by the existing generic `_renderStatBreakdown` popover). The Rakish Audacity branch was removed from the `initiativeBonus` auto-effect emit (`effects.push` in `_processFeatureAutoEffects`) to avoid double-counting; the `case "initiativeBonus":` handler is left in place for any future feat-driven uses. Mirrors the Jack-of-All-Trades pattern already in `getInitiative()` — live re-derivation each call, so ASI / item / level changes propagate immediately. Covered by new `test/jest/charactersheet/CharacterSheetInitiativeBonuses.test.js` (10 tests across 5 describes): Chronurgy +INT total + breakdown + live re-derivation on INT change, Gloom Stalker +WIS, Swashbuckler +CHA with explicit no-double-count assertion (`customModifiers.initiative === 0`) and live re-derivation on CHA change, 2024 Champion +PB, plus stacking with custom modifier and breakdown-total invariants. Existing Rakish Audacity test in `CharacterSheetFeatureEffects.test.js` was rewritten to assert the live-aggregator contract (initiative total + breakdown component) instead of the now-removed `getAppliedClassFeatureEffects` text. Full charsheet suite: 119 / 8159 green.

[x] Players need a way to track buffs cast on them by party members (Aid, Bless, Haste, Heroes' Feast, etc.) without having to be spellcasters themselves — fixed in two layers. (1) HP-calc consumer: `_calculateMaxHp` and `getHpBreakdown` now aggregate `{type: "hpMaxIncrease", value: N}` effects from all currently-active states (via a new `_aggregateActiveStateHpEffects()` helper), and `activateState` / `deactivateState` / `removeActiveState` / `advanceRound` call a gated `_syncCurrentHpToMaxDelta(oldMax)` whenever the affected state actually carries `hpMaxIncrease` — keeping max + current HP in lockstep with RAW (Aid grants both, expiry caps current to the new max but never raises it, never drops below 1). The gating is important: Wild Shape and other states that manage HP out-of-band must not trigger a recalculation. This incidentally also fixes Aid cast by spellcasters on themselves, where the `aid` entry in `SPELL_BUFF_REGISTRY` had been shipping `hpMaxIncrease` for a long time without any consumer applying it. (2) Combat-tab Apply Buff picker: a new `✨ Apply Buff` button in the Quick Actions row opens a search-filtered modal listing the full `SPELL_BUFF_REGISTRY` (40+ spells), each with a one-line effect summary, duration / concentration label, and — for spells with numeric `upcastPerLevel` scaling (e.g. Aid, Armor of Agathys, False Life) — a "Slots above base" input that scales `hpMaxIncrease` and `tempHp` accordingly. On apply, the picker mirrors the activateState payload built by the spell-cast flow in `charactersheet-spells.js` so all downstream consumers (HP, AC, attacks, advantage, etc.) treat picker-applied buffs identically. Stacking-rejection: if a spell-effect state with the same name is already active, the picker refuses with a toast — the user must toggle the existing one off first to "re-cast"; the calc layer is also defensively summation-safe in case the gate is ever bypassed. Covered by `test/jest/charactersheet/CharacterSheetSpellBuffEffects.test.js` (14 tests): Aid base activation bumping max + current, mid-fight activation only adding the delta, current-HP cap on activation/deactivation, breakdown row appearance, expiry never dropping current below 1, upcast at slot 3 (+10), stacking sums across distinct sources, deactivated states ignored, `removeActiveState` re-caps, `advanceRound` mid-combat expiry recomputes max, and the picker stacking-rejection contract on `getActiveStates()`.
[x] Celestial Resistance of Hochling/Aasimar isn't appearing as a resistance — fixed at the data layer. Hochling is a TGTT race that uses `_copy: {name: "Aasimar", source: "MPMM"}` to inherit everything (including `resist: ["necrotic", "radiant"]`) from MPMM Aasimar. Although `homebrew/TravelersGuidetoThelemar.json` correctly declared `_meta.dependencies.race` (so MPMM races are loadable as copy targets), it never declared `_meta.internalCopies`, so the brew loader's `pDoMetaMerge` left in-file `_copy` directives unresolved — Hochling reached `CharacterSheetBuilder._applyRacialTraits()` as `{name, source, alias, _copy}` with no `resist` field, the builder's `if (race.resist)` branch was undefined, and no resistance was added to state. Official Aasimar (DMG / MPMM / VGM / XPHB) declares `resist` directly and was never affected — the bug title conflated the two. Fix: added `"internalCopies": ["spell", "class", "subclass", "race"]` to TGTT's `_meta` (audit confirmed those four arrays are the ones with in-file `_copy` directives — 22 spells, 1 class, 159 subclasses, 16 races — so the same fix also unblocks any other silently-broken TGTT in-file copies, e.g. the Goodberry override and the Hochling/Tengu/Kobold/etc. race chain). Covered by `test/jest/charactersheet/CharacterSheetHochlingResistance.test.js` (6 tests): JSON invariants on the homebrew file (`internalCopies` declared, MPMM listed in race deps, Hochling has the expected `_copy`, MPMM Aasimar still carries the resistances), plus a builder-ingestion smoke test with both a positive branch (merged race ⇒ resistances appear on state) and a negative branch (unmerged race ⇒ no resistances — proves the spec would have caught the original bug). Note: the comprehensive E2E spec `tgtt-child-of-sun-sorcerer-hochling.spec.ts` has resistance probes nested under a Sorcery Points feature entry that is currently skipped for unrelated `CS-BUG-018` (TGTT class resource maxes); those probes will start exercising automatically when CS-BUG-018 is resolved.
