[x] Dauntless Heritage was being applied to all saves regardless of trigger, surfacing the deeper architectural question of how to handle conditional modifiers in the sheet — fixed with a single unified system for both flavors of conditional ("conditional under which a modifier applies") that previously coexisted but worked in parallel and inconsistently. **Two latent encodings, one fix.** (1) The TGTT trait parser at [js/charactersheet/charactersheet-state.js](js/charactersheet/charactersheet-state.js#L1143) (`advantageSavePatterns`) was emitting `{type: "save:all", advantage: true, conditional: "against being frightened"}` for any "advantage on saves against being/against X" prose. The aggregator ignored the `conditional` field entirely, so Dauntless Heritage gave the character advantage on *every* save. (2) Hand-authored registry entries used the parallel sub-type encoding `{type: "save:advantage:frightened"}` — which `_parseModifierType` correctly stripped to baseType `"save:frightened"`, but `getModifiersForType("save:dex")` never matched (only basic abilities or `:all`), so those entries were silently invisible at roll time even when correctly authored. **Architectural decision: per-roll opt-in via a pre-roll picker.** Default-off gating is the only sound default — auto-applying conditional advantage to every roll is the bug, but auto-applying *none* and offering them in a picker preserves discoverability and lets the player make the in-fiction judgment ("is this attacker actually frightening you?") that no static rule can. Toggle-style "this condition is currently true on me" auto-application was considered and rejected as a v2 — it doesn't compose well across mixed conditional sources (frightened-vs-spells-vs-disease) and would force every parsed conditional to round-trip through a registry of condition keys. **Aggregator (Phase A).** `aggregateModifiers(type, opts?)` at [charactersheet-state.js#L28575](js/charactersheet/charactersheet-state.js#L28575) now takes an optional `{appliedConditionalIds: Set<string>}` second argument; any mod with a truthy `conditional` field is skipped from the contribution path and surfaced in a new `result.conditionalsAvailable` array (`{id, name, conditional, advantage, disadvantage, bonus, target}`) unless its stable ID is in the opt-in set. Stable ID is `"${baseType}|${name||note||""}|${conditional}"` — deterministic across repeated aggregations within a roll cycle. `getAdvantageState(type, opts?)` and `getModifierBonus(type, opts?)` forward the opt unchanged. Two new pure static helpers on `CharacterSheetState`: `_isConditionalSaveSubtype(subtype)` (rejects basic abilities + `"all"` + standard skill names; accepts `frightened`/`charmed`/`poisoned`/`magic`/`disease`/`spells`/…) and `_buildConditionalModId(mod)`. **Registry normalization (Phase A, same file).** `getModifiersForType` was extended so that when the query is `save:<ability>` / `check:<ability>` / `skill:<key>` and a registry entry exists with a conditional sub-type, that entry now *also* matches the query — with a synthesized `conditional` string ("against being frightened", "against spells", etc., with the `being` prefix reserved for the actual conditions you can be afflicted with) so it flows through the same opt-in path as text-parsed conditionals. Net effect: both encodings produce the same picker UX and the same audit trail. **Picker UX (Phase B).** New `_pPickConditionalModifiers({rollLabel, conditionalsAvailable})` on `CharacterSheetPage` at [charactersheet.js#L8929](js/charactersheet/charactersheet.js#L8929) — a single-purpose modal with one checkbox row per available conditional, color-coded chips (`Advantage` / `Disadvantage` / `+N`), `Apply all` / `Apply none` bulk buttons, and three actions: `Cancel roll` (aborts entirely), `Roll without` (skip all), `Apply selected & Roll` (commits the picked subset). Returns `{appliedConditionalIds, applied, cancelled}`. Skipped silently when the available list is empty (zero noise on normal rolls) or when the player has set `settings.skipConditionalPrompt` (escape hatch — new checkbox in the existing dice-settings dropdown at [charactersheet.html#L256](charactersheet.html#L256), wired via `_initDicePicker`). Wired into all four d20 roll sites: `_rollAbilityCheck` ([L8957](js/charactersheet/charactersheet.js#L8957)), `_rollSavingThrow` ([L9060](js/charactersheet/charactersheet.js#L9060)), `_rollSkillCheck` ([L9150](js/charactersheet/charactersheet.js#L9150) — probes both `skill:X` and `check:<ability>` pools and dedupes by ID), and `_rollAttack` ([L9350](js/charactersheet/charactersheet.js#L9350) — converted to `async`; only the *opted-in* conditional numeric bonus is added on top of `attack.attackBonus` to avoid double-counting registry mods already folded into the attack-building pipeline). Each roll site emits a `⚡ {Name} ({effect}, {conditional})` line per applied conditional via a tiny `_formatAppliedConditionalsNote` helper so the dice-result reminder shows exactly which conditionals contributed. **Styling.** Picker SCSS appended to [css/charactersheet.css](css/charactersheet.css#L17648) (`.charsheet__cond-pick*` family — rows with hover, color-coded chips with adv-green / dis-red / generic-indigo variants, max-height scroll, full night-mode parity). **Out of scope (recorded for future):** sticky "this conditional is always on for me" toggles, conditional resistances, auto-detection from currently-active conditions on self, bulk migration of hand-authored registry entries to the new shape (already covered transparently by the normalization layer). **Tests:** new [test/jest/charactersheet/CharacterSheetConditionalModifiers.test.js](test/jest/charactersheet/CharacterSheetConditionalModifiers.test.js) (15 tests across 3 describes) pins the helper contracts (basic abilities + `"all"` + skill names are not conditional; conditions / damage types / "magic" / "disease" / "spells" are; stable ID determinism and `:advantage:` stripping), the gating contract (text-parsed Dauntless Heritage doesn't auto-apply to any of the 6 ability saves; opt-in via stable ID does apply it; non-conditional `save:all` modifiers still auto-apply unchanged; registry sub-typed `save:advantage:frightened` is surfaced as a conditional on `save:<ability>` queries with the right synthesized text; opt-in works for both numeric bonuses and adv flags; identical mods dedupe to one picker row), and the opt-forwarding contract (`getAdvantageState` + `getModifierBonus` both honour `appliedConditionalIds`). Existing TGTT parser tests for Pious Soul / Stable Footing pass unchanged (they only assert the modifier *exists* with conditional text, which still holds). Returns `{added, classesProcessed}` for logging, wrapped in try/catch for safety. **(3) Roll-note brevity.** Added a `shortLabel` field to the passive-defense alert objects returned by `getPassiveSaveAlerts(ability)` in [js/charactersheet/charactersheet-state.js](js/charactersheet/charactersheet-state.js#L6939) — Evasion gets `"save → 0 dmg, fail → ½"`, Last Ditch Evasion gets `"Reaction: fail → ½ instead of full"`. The dex-save reminder line in `_rollSavingThrow` now renders `💡 ${a.name} (${a.shortLabel || a.summary})` so players see a one-liner instead of three sentences of rules text; full text still lives on the feature card. JSDoc updated to document the new field. [x] Nyuidj from TGTT have advantage on Wis saves (Dual Mind racial trait, registered as a `save:advantage:wis` modifier), but pressing Ctrl/Cmd to add disadvantage rolled with disadvantage instead of canceling to a normal roll per RAW. Root cause was in `_rollD20` in `js/charactersheet/charactersheet.js` (~L8500): the four roll callsites (saves, ability checks, skill checks, attacks) precomputed a `mode` string from passive state alone, then `_rollD20` *unconditionally* overrode that mode with the event modifier keys (`shiftKey` → `"advantage"`, `ctrlKey/metaKey` → `"disadvantage"`) — so a passive `mode = "advantage"` silently became `"disadvantage"` on Ctrl, with no chance to cancel. Symmetric defect: passive disadvantage + Shift also stacked instead of canceling. Fixed by extracting a single pure helper `CharacterSheetClassUtils.resolveD20Mode({stateAdvantage, stateDisadvantage, event})` that combines the two sources symmetrically — `adv = stateAdvantage || event?.shiftKey`, `dis = stateDisadvantage || event?.ctrlKey || event?.metaKey`, then `adv && dis → "normal"`, else single side wins. `_rollD20` now takes optional `stateAdvantage`/`stateDisadvantage` booleans; when either is provided it routes through the helper (cancel-aware path), otherwise it preserves the legacy event-only path so `_rollInitiative` and the public `rollD20` pass-through API are unchanged. The four affected callsites — `_rollAbilityCheck`, `_rollSavingThrow`, `_rollSkillCheck`, `_rollAttack` — were updated to drop their local `let mode; if (hasAdv && !hasDis) mode = …` blocks and pass the booleans directly, and each now also computes an effective `(stateAdvantage || event.shiftKey, stateDisadvantage || event.ctrlKey/metaKey)` pair for `_getActiveStateEffectLabel` so the dice-result title shows `[States: Adv+Disadv cancel]` whenever cancellation actually fires (including the previously invisible "user keys cancel passive state" case). Helper lives on `CharacterSheetClassUtils` so tests don't need to load the full DOM-coupled `CharacterSheetPage` module. Covered by new `test/jest/charactersheet/CharacterSheetD20ModeResolution.test.js` (16 tests across one describe): every (stateAdv, stateDis, shiftKey, ctrlKey, metaKey) combination that matters — empty input, state-only adv/dis, event-only adv/dis (shift, ctrl, meta), the Nyuidj fix (state adv + ctrl → normal), the Mac variant (state adv + meta → normal), the symmetric case (state disadv + shift → normal), agreeing-stack cases (state adv + shift stays adv; state disadv + ctrl stays disadv), three-way cancellation (both state sides + shift still cancels), explicit-false equivalence, and an irrelevant-key probe (alt key ignored). The 19 existing TGTT Nyuidj tests pass unchanged. Full charsheet suite: 122 / 8185 green.
